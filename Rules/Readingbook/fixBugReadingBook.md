# ReadingGraphApp 修复bug流程

【铁律 - Bug 修复流程（不可违背）】
1. 绝对禁止猜测问题原因。必须先复现 + 收集证据（日志、调用栈、输入节点数量、性能指标、报错截图）。
2. 任何涉及「大量节点的关系图」bug，必须先执行「业界调研技能」，不得直接写修复代码。
3. 调研后必须输出：
   - 业界成熟方案对比表（至少3个主流库/论文/开源项目）
   - 为什么当前实现有缺陷（根本原因）
   - 通用化设计方案（不止解决当前case，要覆盖 1000~10万节点量级）
4. 最终 PR/修改必须包含：
   - 通用抽象层（而非 if-case 补丁）
   - 可扩展的配置参数
   - 性能基准测试前后对比
5. 若 AI 发现自己无法调研或证据不足，必须回复「需要更多上下文/日志」并停止修复。
